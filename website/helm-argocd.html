⛵ Partie A — Déploiement avec Helm

Packager l'application dans un Helm Chart réutilisable et configurable.

H1
Comprendre la structure d'un Helm Chart
Analyser l'architecture d'un chart Helm et le rôle de chaque fichier.

🎯 Tâches à réaliser

  • Créer le dossier helm/demo-app/
  • Créer le fichier Chart.yaml avec les métadonnées du chart
  • Créer le fichier values.yaml avec les valeurs par défaut
  • Créer le dossier templates/ pour les manifests templatisés

📖 Ce que vous devez savoir

Un Helm Chart est un package Kubernetes. Il contient des templates YAML paramétrables via values.yaml. Helm génère les manifests finaux en injectant les valeurs dans les templates Go.

helm/demo-app/
├── Chart.yaml          # Nom, version, description
├── values.yaml         # Valeurs par défaut
└── templates/
    ├── _helpers.tpl    # Fonctions réutilisables
    ├── namespace.yaml
    ├── configmap.yaml
    ├── secret.yaml
    ├── persistence.yaml  # PV + PVC
    ├── deployment.yaml
    ├── service.yaml
    ├── ingress.yaml
    └── NOTES.txt       # Message post-install
💡 Indice : Chart.yaml doit contenir apiVersion: v2, name, version, et appVersion.
H2
Créer le fichier values.yaml
Définir toutes les valeurs configurables : image, replicas, config, secrets, ingress, etc.

🎯 Tâches à réaliser

  • Définir les paramètres de l'image Docker (repository, tag, pullPolicy)
  • Configurer le nombre de replicaCount
  • Définir les variables du ConfigMap sous config:
  • Définir les secrets sous secret: (valeurs en base64)
  • Configurer le Service (type, port, targetPort)
  • Configurer l'Ingress (enabled, host, className)
  • Définir les resources (requests/limits) et les probes
  • Ajouter des options autoscaling et networkPolicy (désactivés par défaut)

📖 Ce que vous devez savoir

Le fichier values.yaml est la source unique de configuration. L'utilisateur peut surcharger ces valeurs avec --set ou -f custom-values.yaml lors de l'installation.

💡 Indice : Utilisez des blocs conditionnels : autoscaling.enabled: false pour activer/désactiver des fonctionnalités sans supprimer du code.
helm lint helm/demo-app/
H3
Écrire les templates Kubernetes
Transformer les manifests YAML statiques en templates Helm dynamiques.

🎯 Tâches à réaliser

  • Créer _helpers.tpl avec les fonctions : demo-app.name, demo-app.fullname, demo-app.labels, demo-app.selectorLabels
  • Templatiser chaque manifest en utilisant {{ .Values.xxx }}
  • Utiliser {{ include "demo-app.fullname" . }} pour les noms
  • Ajouter {{- if .Values.xxx.enabled }} pour les ressources optionnelles (Ingress, HPA, NetworkPolicy)
  • Dans le Deployment, ajouter des annotations checksum/config pour le redéploiement automatique

📖 Ce que vous devez savoir

Les templates Helm utilisent le moteur Go template. Les fonctions clés : {{ .Values }} pour les valeurs, {{ include }} pour les helpers, {{- toYaml }} pour sérialiser des blocs, {{- if }} pour les conditions.

💡 Indice : L'annotation checksum/config: {{ include (print $.Template.BasePath "/configmap.yaml") . | sha256sum }} force le redéploiement quand la config change.
H4
Installer et tester le chart Helm
Déployer l'application avec Helm et vérifier le résultat.

🎯 Tâches à réaliser

  • Valider la syntaxe : helm lint helm/demo-app/
  • Prévisualiser les manifests : helm template demo-app helm/demo-app/
  • Installer le chart : helm install demo-app helm/demo-app/
  • Vérifier le déploiement : kubectl get all -n demo-app
  • Tester un override : helm upgrade demo-app helm/demo-app/ --set replicaCount=3
  • Voir l'historique : helm history demo-app
# Commandes Helm essentielles
helm lint helm/demo-app/                     # Valider
helm template demo-app helm/demo-app/        # Prévisualiser
helm install demo-app helm/demo-app/         # Installer
helm upgrade demo-app helm/demo-app/         # Mettre à jour
helm rollback demo-app 1                     # Rollback
helm uninstall demo-app                      # Supprimer
helm list                                     # Lister les releases
helm list
kubectl get all -n demo-app

🔄 Partie B — Déploiement GitOps avec ArgoCD

Mettre en place un pipeline de déploiement continu et déclaratif.

A1
Installer ArgoCD sur le cluster
Déployer ArgoCD dans un namespace dédié et accéder à l'interface web.

🎯 Tâches à réaliser

  • Créer le namespace argocd
  • Installer ArgoCD avec les manifests officiels
  • Attendre que tous les pods soient prêts
  • Récupérer le mot de passe admin initial
  • Accéder à l'UI via port-forward

📖 Ce que vous devez savoir

ArgoCD est un outil de déploiement continu GitOps pour Kubernetes. Il surveille un repo Git et synchronise automatiquement l'état du cluster avec l'état déclaré dans le code.

# Installer ArgoCD
kubectl create namespace argocd
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

# Attendre que ce soit prêt
kubectl wait --for=condition=available deployment/argocd-server -n argocd --timeout=120s

# Mot de passe admin
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d

# Accéder à l'UI
kubectl port-forward svc/argocd-server -n argocd 8080:443
# → Ouvrir https://localhost:8080 (user: admin)
kubectl get pods -n argocd
→ Tous les pods doivent être Running
A2
Créer un AppProject ArgoCD
Isoler les applications du TP dans un projet dédié avec des restrictions.

🎯 Tâches à réaliser

  • Créer un fichier argocd/project.yaml
  • Définir un AppProject nommé tp-kubernetes
  • Restreindre les sourceRepos à votre repo GitHub
  • Autoriser uniquement le namespace demo-app comme destination
  • Définir les types de ressources autorisés (Deployment, Service, Ingress, etc.)

📖 Ce que vous devez savoir

Un AppProject isole les applications ArgoCD. Il restreint les repos sources, les namespaces cibles, et les types de ressources autorisés. C'est une bonne pratique de sécurité.

💡 Indice : Les PV et Namespace sont des clusterResourceWhitelist (pas dans un namespace), le reste est dans namespaceResourceWhitelist.
kubectl get appproject -n argocd
A3
Créer l'Application ArgoCD (manifests bruts)
Déployer l'application en pointant directement sur le dossier k8s/.

🎯 Tâches à réaliser

  • Créer un fichier argocd/application-manifests.yaml
  • Définir une Application ArgoCD pointant vers le dossier k8s/
  • Configurer la source : repoURL, targetRevision (main), path (k8s)
  • Configurer la destination : server (cluster local), namespace
  • Activer le syncPolicy.automated avec prune: true et selfHeal: true
  • Ajouter le finalizer pour le nettoyage automatique

📖 Ce que vous devez savoir

  • prune: true → Supprime les ressources K8s qui n'existent plus dans Git
  • selfHeal: true → Corrige automatiquement les modifications manuelles faites sur le cluster
  • finalizer → Quand on supprime l'Application ArgoCD, les ressources K8s sont aussi supprimées
💡 Indice : Le repoURL doit être en HTTPS pour un repo public. Ajoutez CreateNamespace=true dans les syncOptions.
kubectl get applications -n argocd
A4
Créer l'Application ArgoCD (Helm chart)
Déployer l'application via le Helm chart stocké dans le repo Git.

🎯 Tâches à réaliser

  • Créer un fichier argocd/application-helm.yaml
  • Définir une Application avec le path vers helm/demo-app
  • Configurer la section source.helm avec :
    • releaseName : nom de la release Helm
    • valueFiles : liste des fichiers de valeurs
    • parameters : override de valeurs (optionnel)
  • Appliquer la même syncPolicy que pour les manifests bruts

📖 Ce que vous devez savoir

ArgoCD détecte automatiquement qu'un dossier contient un Chart.yaml et active le mode Helm. On peut override les values directement dans l'Application ou pointer vers un fichier de valeurs.

💡 Indice : ArgoCD supporte les paramètres Helm : parameters: [{name: "replicaCount", value: "3"}] pour surcharger les valeurs sans modifier values.yaml.
A5
Tester le workflow GitOps complet
Valider le cycle : modifier le code → push → ArgoCD synchronise automatiquement.

🎯 Tâches à réaliser

  • Appliquer le projet et l'application ArgoCD
  • Vérifier dans l'UI ArgoCD que l'application est "Synced" et "Healthy"
  • Modifier values.yaml (ex: changer replicaCount de 2 à 3)
  • git commit et git push
  • Observer ArgoCD détecter et appliquer le changement automatiquement
  • Tester le self-heal : modifier manuellement un Deployment avec kubectl et observer ArgoCD le corriger
# 1. Déployer avec ArgoCD
kubectl apply -f argocd/project.yaml
kubectl apply -f argocd/application-helm.yaml

# 2. Vérifier dans ArgoCD
kubectl get applications -n argocd

# 3. Modifier values.yaml → push → ArgoCD sync auto

# 4. Test Self-Heal : modifier manuellement
kubectl scale deployment demo-app -n demo-app --replicas=5
# → ArgoCD va remettre à 2 automatiquement !

# 5. Voir les détails (CLI)
argocd app get demo-app-helm
argocd app diff demo-app-helm
argocd app list
→ STATUS: Synced, HEALTH: Healthy
