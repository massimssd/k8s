1. Namespace
namespace.yaml — Isolation logique du projet
Qu'est-ce que c'est ? Un Namespace crée un espace de noms logique dans le cluster. Toutes nos ressources (Deployment, Service, etc.) vivent dans ce namespace, isolées du reste du cluster.

Pourquoi ? Permet d'organiser les ressources, d'éviter les conflits de noms, et de faciliter la gestion des droits d'accès (RBAC). 
# namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: demo-app
  labels:
    app: k8s-demo
    env: demo
    author: khlifi-houcem
ChampExplication
kind: NamespaceType d'objet Kubernetes
metadata.nameNom unique du namespace dans le cluster
metadata.labelsÉtiquettes pour identifier et filtrer les ressources

2. ConfigMap
configmap.yaml — Variables de configuration non sensibles
Qu'est-ce que c'est ? Un ConfigMap stocke des données de configuration sous forme clé-valeur. Il permet de séparer la configuration du code applicatif.

Pourquoi ? Modifier la configuration sans reconstruire l'image Docker. On peut changer APP_NAME ou LOG_LEVEL et simplement redémarrer les pods. 
# configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: demo-app-config
  namespace: demo-app
  labels:
    app: k8s-demo
data:
  APP_NAME: "K8s Demo Notes"
  APP_ENV: "production"
  APP_VERSION: "1.0.0"
  LOG_LEVEL: "info"
  STORAGE_PATH: "/data"
  FEATURE_UPLOAD_ENABLED: "true"
  PORT: "3000"
ChampExplication
metadata.namespaceLe ConfigMap vit dans le namespace demo-app
dataPaires clé-valeur (toujours des strings)
STORAGE_PATH: "/data"Chemin où le PVC sera monté dans le container
FEATURE_UPLOAD_ENABLEDFeature flag contrôlable sans rebuild

3. Secret
secret.yaml — Données sensibles encodées en base64
Qu'est-ce que c'est ? Un Secret est similaire à un ConfigMap mais conçu pour les données sensibles (mots de passe, clés API, tokens). Les valeurs sont encodées en base64.

Pourquoi ? Séparer les secrets du code source. En production, on utilise un outil comme Sealed Secrets ou Vault pour un chiffrement réel. ⚠️ base64 n'est PAS du chiffrement ! 
# secret.yaml
# Valeurs en clair pour référence :
#   APP_SECRET: "mon-super-secret-k8s-2024"
#   ADMIN_PASSWORD: "AdminK8s@Secure!"
# Encoder : echo -n "valeur" | base64
apiVersion: v1
kind: Secret
metadata:
  name: demo-app-secret
  namespace: demo-app
  labels:
    app: k8s-demo
type: Opaque
data:
  APP_SECRET: bW9uLXN1cGVyLXNlY3JldC1rOHMtMjAyNA==
  ADMIN_PASSWORD: QWRtaW5LOHNAU2VjdXJlIQ==
ChampExplication
type: OpaqueType générique pour les secrets arbitraires
dataValeurs encodées en base64 (pas en clair !)
APP_SECRETClé utilisée pour signer les tokens HMAC-SHA256 après le login
ADMIN_PASSWORDMot de passe admin, vérifié par POST /api/login

3b. Utilisation des Secrets dans l'Application
Authentification, Login et Tokens signés
Comment les Secrets sont utilisés concrètement ?

🔑 ADMIN_PASSWORD → L'utilisateur clique sur "🔐 Connexion Admin" dans l'interface. Le backend vérifie le mot de passe contre la variable ADMIN_PASSWORD injectée depuis le Secret K8s.

🔒 APP_SECRET → Après un login réussi, un token est généré et signé avec HMAC-SHA256 en utilisant APP_SECRET. Ce token est requis pour les opérations de suppression.

🛡️ Protection → Sans authentification, les boutons de suppression affichent un 🔒 et les requêtes DELETE retournent une erreur 401. 
# Backend (server.js) — Login avec ADMIN_PASSWORD
app.post('/api/login', (req, res) => {
  const { username, password } = req.body;
  // ADMIN_PASSWORD provient du Secret Kubernetes
  if (password !== ADMIN_PASSWORD) {
    return res.status(401).json({ error: 'Mot de passe incorrect' });
  }
  // Générer un token signé avec APP_SECRET (du Secret K8s)
  const token = generateToken(username);
  res.json({ message: 'Connexion réussie', user: username, token });
});

# Génération de token HMAC-SHA256 avec APP_SECRET
function generateToken(username) {
  const payload = JSON.stringify({
    user: username,
    iat: Date.now(),
    exp: Date.now() + 3600000, // 1 heure
  });
  const payloadB64 = Buffer.from(payload).toString('base64');
  const signature = crypto
    .createHmac('sha256', APP_SECRET)  // ← Secret K8s !
    .update(payloadB64)
    .digest('hex');
  return `${payloadB64}.${signature}`;
}

# Routes protégées par le token
app.delete('/api/notes/:id', requireAuth, (req, res) => {
  // requireAuth vérifie le token signé avec APP_SECRET
  // ...
});
# Test depuis le terminal

# 1. Login (utilise ADMIN_PASSWORD du Secret)
curl -X POST http://demo.local/api/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"AdminK8s@Secure!"}'
# → Retourne un token signé avec APP_SECRET

# 2. Supprimer sans token (REJETÉ - 401)

# → {"error": "Authentification requise"}

# 3. Supprimer avec token (ACCEPTÉ)

# → {"message": "Note supprimée"}

4. PersistentVolume
pv.yaml — Stockage physique sur le nœud
Qu'est-ce que c'est ? Un PV est une ressource de stockage dans le cluster, provisionnée par l'administrateur. Il représente un espace disque réel.

Pourquoi ? Sans PV, les données sont perdues quand un pod redémarre. Le PV garantit la persistance des notes et fichiers uploadés. ⚠️ Le PV n'a PAS de namespace. 
# pv.yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: demo-app-pv
  labels:
    app: k8s-demo
    type: local
spec:
  capacity:
    storage: 1Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: demo-storage
  hostPath:
    path: /tmp/demo-app-data
    type: DirectoryOrCreate
ChampExplication
capacity.storageTaille du volume : 1 Gigaoctet
ReadWriteOnceUn seul nœud peut monter le volume en lecture-écriture
RetainLes données sont conservées même si le PVC est supprimé
storageClassNameFait le lien avec le PVC (doit correspondre)
hostPathUtilise un dossier du nœud (démo seulement, pas en prod !)

5. PersistentVolumeClaim
pvc.yaml — Demande de stockage par l'application
Qu'est-ce que c'est ? Un PVC est une demande de stockage faite par un pod. Kubernetes lie automatiquement le PVC au PV compatible.

Pourquoi ? Le PVC est l'interface entre le pod et le stockage. Le Deployment référence le PVC (pas le PV directement), ce qui découple l'application du stockage physique. 
# pvc.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: demo-app-pvc
  namespace: demo-app
  labels:
    app: k8s-demo
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: demo-storage
  resources:
    requests:
      storage: 1Gi
ChampExplication
namespace: demo-appLe PVC est dans le namespace (contrairement au PV)
storageClassNameDoit correspondre au PV pour le binding automatique
resources.requestsQuantité de stockage demandée (≤ capacité du PV)

6. Deployment
deployment.yaml — Cœur du déploiement applicatif
Qu'est-ce que c'est ? Le Deployment gère le cycle de vie des pods : création, mise à jour, scaling, rollback. C'est l'objet le plus important du déploiement.

Pourquoi ? Il orchestre tout : nombre de replicas, injection du ConfigMap et Secret, montage du PVC, probes de santé, limites de ressources, et stratégie de mise à jour. 
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: demo-app
  namespace: demo-app
  labels:
    app: k8s-demo
    version: v1
spec:
  replicas: 2
  selector:
    matchLabels:
      app: k8s-demo
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  template:
    metadata:
      labels:
        app: k8s-demo
        version: v1
    spec:
      containers:
        - name: demo-app
          image: k8s-demo-app:latest
          imagePullPolicy: IfNotPresent
          ports:
            - containerPort: 3000
              name: http

          # Variables depuis ConfigMap (toutes les clés)
          envFrom:
            - configMapRef:
                name: demo-app-config

          # Variables depuis Secret (clé par clé)
          env:
            - name: APP_SECRET
              valueFrom:
                secretKeyRef:
                  name: demo-app-secret
                  key: APP_SECRET
            - name: ADMIN_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: demo-app-secret
                  key: ADMIN_PASSWORD

          # Montage du volume persistant
          volumeMounts:
            - name: data-volume
              mountPath: /data

          # Limites de ressources
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
            limits:
              cpu: 500m
              memory: 256Mi

          # Probe : le pod est-il prêt à recevoir du trafic ?
          readinessProbe:
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 5
            periodSeconds: 10
            timeoutSeconds: 3
            failureThreshold: 3

          # Probe : le pod est-il toujours vivant ?
          livenessProbe:
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 15
            periodSeconds: 20
            timeoutSeconds: 5
            failureThreshold: 3

      volumes:
        - name: data-volume
          persistentVolumeClaim:
            claimName: demo-app-pvc

      restartPolicy: Always
      terminationGracePeriodSeconds: 30
ChampExplication
replicas: 22 instances du pod pour la haute disponibilité
selector.matchLabelsLe Deployment gère les pods ayant ce label
RollingUpdateMise à jour progressive sans interruption
envFrom.configMapRefInjecte TOUTES les clés du ConfigMap comme variables d'env
env.valueFrom.secretKeyRefInjecte UNE clé spécifique du Secret
volumeMounts.mountPathLe PVC est monté sur /data dans le container
resources.requestsMinimum garanti de CPU/RAM pour le scheduling
resources.limitsMaximum autorisé (le pod est tué s'il dépasse)
readinessProbeVérifie que le pod peut servir du trafic (ajoute/retire des endpoints)
livenessProbeVérifie que le pod est vivant (le redémarre s'il échoue)
imagePullPolicy: IfNotPresentNe pull pas l'image si elle existe déjà localement (Minikube)

7. Service
service.yaml — Exposition interne et load-balancing
Qu'est-ce que c'est ? Un Service fournit une IP stable et un nom DNS interne pour accéder aux pods. Il répartit le trafic entre les replicas.

Pourquoi ? Les pods sont éphémères (IPs changent). Le Service offre un point d'accès stable : demo-app-service.demo-app.svc.cluster.local 
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: demo-app-service
  namespace: demo-app
  labels:
    app: k8s-demo
spec:
  type: ClusterIP
  selector:
    app: k8s-demo
  ports:
    - name: http
      protocol: TCP
      port: 80
      targetPort: 3000
ChampExplication
type: ClusterIPAccessible uniquement depuis l'intérieur du cluster
selectorCible les pods avec le label app: k8s-demo
port: 80Port d'écoute du Service (côté client)
targetPort: 3000Port réel de l'application dans le container

8. Ingress
ingress.yaml — Point d'entrée HTTP externe
Qu'est-ce que c'est ? L'Ingress expose les services HTTP/HTTPS à l'extérieur du cluster. Il utilise un contrôleur (nginx) pour router le trafic selon les règles.

Pourquoi ? Permet d'accéder à l'application via http://demo.local depuis le navigateur, au lieu d'utiliser un port-forward. 
# ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: demo-app-ingress
  namespace: demo-app
  labels:
    app: k8s-demo
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
    nginx.ingress.kubernetes.io/proxy-body-size: "10m"
spec:
  ingressClassName: nginx
  rules:
    - host: demo.local
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: demo-app-service
                port:
                  number: 80
ChampExplication
ingressClassNameSélectionne le contrôleur Ingress à utiliser (nginx)
rules[].hostNom de domaine qui route vers le service
pathType: PrefixToutes les URLs commençant par / sont routées
backend.serviceLe Service cible avec son port
annotationsConfiguration spécifique au contrôleur nginx

Bonus : Job d'initialisation
job-init.yaml — Seed de données de démonstration
Qu'est-ce que c'est ? Un Job exécute une tâche ponctuelle (contrairement au Deployment qui tourne en continu). Il se termine quand la tâche est accomplie.

Pourquoi ? Initialiser des données de démo en appelant l'API de l'application. Utile pour peupler la base au premier déploiement. 
# job-init.yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: demo-app-init-data
  namespace: demo-app
spec:
  ttlSecondsAfterFinished: 300
  backoffLimit: 3
  template:
    spec:
      restartPolicy: OnFailure
      containers:
        - name: init-data
          image: curlimages/curl:latest
          command: ["/bin/sh", "-c"]
          args:
            - |
              # Attendre que l'app soit prête
              for i in $(seq 1 30); do
                curl -sf http://demo-app-service/health && break
                sleep 2
              done
              # Créer des notes d'exemple
              curl -X POST http://demo-app-service/api/notes \
                -H "Content-Type: application/json" \
                -d '{"title":"Bienvenue","content":"App déployée sur K8s!","category":"important"}'
ChampExplication
ttlSecondsAfterFinishedLe Job est nettoyé automatiquement après 5 minutes
backoffLimit: 3Maximum 3 tentatives en cas d'échec
restartPolicy: OnFailureRedémarre le container uniquement en cas d'erreur

    - group: ""

      kind: ConfigMap
    - group: ""
      kind: Secret

    - group: "networking.k8s.io"

application-helm.yaml
Application ArgoCD via Helm
ArgoCD détecte Chart.yaml et active le mode Helm. prune: true supprime les ressources obsolètes. selfHeal: true corrige les modifications manuelles (kubectl scale, edit...). 
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: demo-app-helm
  namespace: argocd
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: default
  source:
    repoURL: https://github.com/houcemdevops007/TP-K8s-demo.git
    targetRevision: main
    path: helm/demo-app
    helm:
      releaseName: demo-app
      valueFiles:
        - values.yaml
  destination:
    server: https://kubernetes.default.svc
    namespace: demo-app
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
    retry:
      limit: 3
      backoff:
        duration: 5s
        factor: 2
        maxDuration: 3m
ChampExplication
source.pathChemin vers le chart dans le repo
source.helm.releaseNameNom de la release Helm
prune: trueSupprime les ressources retirées de Git
selfHeal: trueCorrige les modifications manuelles
finalizersNettoyage auto à la suppression
retry.backoffRetry exponentiel (5s → 10s → 20s)

kubectl apply -n argocd -f \
  https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
kubectl wait --for=condition=available \
  deployment/argocd-server -n argocd --timeout=120s

# Mot de passe admin
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d

# UI : https://localhost:8080 (user: admin)
kubectl port-forward svc/argocd-server -n argocd 8080:443

# ── Déployer ─────────────────────────────
kubectl apply -f argocd/project.yaml
kubectl apply -f argocd/application-helm.yaml

# ── Vérifier ─────────────────────────────
kubectl get applications -n argocd
argocd app list
argocd app get demo-app-helm

# ── Test Self-Heal ───────────────────────
kubectl scale deployment demo-app -n demo-app --replicas=5
# → ArgoCD remet à 2 automatiquement !

# ── Test GitOps ──────────────────────────
# 1. Modifier values.yaml → replicaCount: 3
# 2. git commit && git push
# 3. ArgoCD synchronise automatiquement !

# ── Nettoyage ────────────────────────────
kubectl delete -f argocd/application-helm.yaml
kubectl delete -f argocd/project.yaml
