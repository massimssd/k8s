Déploiement d'une Application Complète sur Kubernetes

Dans ce TP, vous allez déployer une application web réelle sur un cluster Kubernetes en utilisant les objets fondamentaux : Namespace, ConfigMap, Secret, PV/PVC, Deployment, Service et Ingress.

🎯 Comprendre les objets K8s
🐳 Containeriser une application
📦 Déployer avec des manifests YAML
🔐 Gérer secrets et configuration
💾 Persister des données
🌐 Exposer via Service et Ingress

📋 Prérequis

Assurez-vous que ces outils sont installés et fonctionnels avant de commencer.

🐳
Docker docker --version
☸
kubectl kubectl version
🖥️
Minikube ou Kind minikube start
📝
Éditeur de texte (VS Code recommandé)

📱 Application à déployer

L'application "K8s Demo Notes" est une app Node.js/Express de gestion de notes avec :

📝
CRUD de notes (créer, lire, modifier, supprimer)
📤
Upload de fichiers
💚
Endpoint /health
⚙️
Endpoint /config et /info
💾
Stockage persistant dans /data
🔐
Authentification admin via Secret K8s (ADMIN_PASSWORD + APP_SECRET)
🔑
Page de login avec token HMAC signé par APP_SECRET
🛡️
Suppression protégée (requiert authentification)

🚀 Étapes du TP

Suivez ces étapes dans l'ordre pour déployer l'application sur Kubernetes.

0
Préparer l'image Docker
Construire l'image Docker de l'application pour pouvoir la déployer dans Kubernetes.

🎯 Tâches à réaliser

  • Explorer le code source dans le dossier app/
  • Lire et comprendre le Dockerfile
  • Construire l'image Docker avec le tag k8s-demo-app:latest
  • Si vous utilisez Minikube, construire l'image dans le contexte Docker de Minikube
💡 Indice : Pour Minikube, utilisez eval $(minikube docker-env) avant le build pour que l'image soit directement disponible dans le cluster.
kubectl — Non applicable à cette étape. Vérifiez avec : docker images | grep k8s-demo
1
Créer un Namespace dédié
Isoler toutes les ressources du TP dans un namespace demo-app.

🎯 Tâches à réaliser

  • Créer un fichier k8s/namespace.yaml
  • Définir un Namespace nommé demo-app
  • Ajouter des labels : app: k8s-demo et env: demo
  • Appliquer le manifest avec kubectl apply -f

📖 Ce que vous devez savoir

Un Namespace est un espace de noms logique dans Kubernetes. Il permet d'isoler les ressources et d'éviter les conflits de noms entre différents projets dans un même cluster.

💡 Indice : L'API version est v1 et le kind est Namespace. C'est l'un des objets les plus simples.
kubectl get namespaces | grep demo-app
2
Créer un ConfigMap
Externaliser les variables de configuration non sensibles de l'application.

🎯 Tâches à réaliser

  • Créer un fichier k8s/configmap.yaml dans le namespace demo-app
  • Stocker les variables suivantes : APP_NAME, APP_ENV, APP_VERSION, LOG_LEVEL, STORAGE_PATH, FEATURE_UPLOAD_ENABLED, PORT
  • Le chemin de stockage doit être /data
  • L'upload doit être activé (true)

📖 Ce que vous devez savoir

Un ConfigMap permet de stocker des données de configuration sous forme clé-valeur. Il découple la configuration du code, permettant de modifier les paramètres sans reconstruire l'image.

💡 Indice : Toutes les valeurs dans un ConfigMap sont des chaînes de caractères. Utilisez la section data: pour les définir.
kubectl get configmap -n demo-app
3
Créer un Secret
Stocker les données sensibles : mot de passe admin et clé de signature des tokens.

🎯 Tâches à réaliser

  • Créer un fichier k8s/secret.yaml dans le namespace demo-app
  • Stocker deux variables sensibles : APP_SECRET et ADMIN_PASSWORD
  • Les valeurs doivent être encodées en base64
  • Le type du Secret doit être Opaque

📖 Ce que vous devez savoir

Un Secret est similaire à un ConfigMap mais conçu pour les données sensibles. Les valeurs sont encodées en base64 (attention : encodé ≠ chiffré).

🔐 Utilisation concrète des Secrets dans l'application

  • ADMIN_PASSWORD → utilisé pour la page de login (POST /api/login). L'utilisateur doit entrer ce mot de passe pour se connecter en tant qu'admin.
  • APP_SECRET → utilisé pour signer les tokens HMAC-SHA256. Après le login, un token signé est généré et requis pour les opérations de suppression.
  • Sans authentification, la suppression de notes et fichiers est bloquée (🔒).
💡 Indice : Pour encoder en base64 : echo -n "ma-valeur" | base64. Contrairement au ConfigMap, les valeurs vont dans data: en base64.
kubectl get secret -n demo-app
kubectl describe secret demo-app-secret -n demo-app
4
Créer un PersistentVolume
Provisionner un espace de stockage persistant pour les données de l'application.

🎯 Tâches à réaliser

  • Créer un fichier k8s/pv.yaml (le PV est au niveau cluster, pas dans un namespace)
  • Définir un PersistentVolume de 1Gi
  • Mode d'accès : ReadWriteOnce
  • Utiliser un hostPath pointant vers /tmp/demo-app-data
  • Définir un storageClassName: demo-storage
  • Politique de réclamation : Retain

📖 Ce que vous devez savoir

Un PersistentVolume (PV) représente un espace de stockage physique dans le cluster. Il est provisionné par l'administrateur et existe indépendamment des pods. Le hostPath utilise un dossier sur le nœud (pour la démo seulement).

💡 Indice : Le PV n'a pas de namespace ! Le storageClassName fait le lien avec le PVC. Utilisez type: DirectoryOrCreate pour le hostPath.
kubectl get pv
5
Créer un PersistentVolumeClaim
Demander l'utilisation du stockage persistant depuis le namespace de l'application.

🎯 Tâches à réaliser

  • Créer un fichier k8s/pvc.yaml dans le namespace demo-app
  • Demander 1Gi de stockage
  • Mode d'accès : ReadWriteOnce
  • Utiliser le même storageClassName: demo-storage que le PV

📖 Ce que vous devez savoir

Un PersistentVolumeClaim (PVC) est une demande de stockage faite par un pod. Kubernetes lie automatiquement le PVC au PV correspondant (matching par storageClassName, capacité et mode d'accès).

💡 Indice : Le PVC est dans un namespace. Le storageClassName doit correspondre exactement à celui du PV.
kubectl get pvc -n demo-app
→ Le STATUS doit être "Bound"
6
Créer le Deployment
Déployer l'application avec 2 replicas, probes de santé, et injection des configurations.

🎯 Tâches à réaliser

  • Créer un fichier k8s/deployment.yaml dans le namespace demo-app
  • Déployer 2 replicas de l'image k8s-demo-app:latest
  • Injecter les variables du ConfigMap via envFrom
  • Injecter les variables du Secret via env avec valueFrom.secretKeyRef
  • Monter le PVC sur le chemin /data
  • Définir des resources (requests : 100m CPU, 128Mi RAM / limits : 500m CPU, 256Mi RAM)
  • Configurer une readinessProbe HTTP sur /health port 3000
  • Configurer une livenessProbe HTTP sur /health port 3000
  • Stratégie de déploiement : RollingUpdate

📖 Ce que vous devez savoir

Le Deployment est l'objet central. Il gère le cycle de vie des pods, les mises à jour, le scaling, et garantit que le nombre souhaité de replicas est toujours actif. Les probes permettent à Kubernetes de savoir si le pod est prêt et vivant.

💡 Indice : Pensez à imagePullPolicy: IfNotPresent pour les images locales. Les probes utilisent httpGet avec le path /health.
kubectl get deployment -n demo-app
kubectl get pods -n demo-app -o wide
→ 2 pods doivent être Running
7
Créer un Service ClusterIP
Exposer l'application en interne pour le load-balancing entre les replicas.

🎯 Tâches à réaliser

  • Créer un fichier k8s/service.yaml dans le namespace demo-app
  • Type : ClusterIP
  • Sélecteur : app: k8s-demo
  • Port d'écoute : 80 → targetPort : 3000

📖 Ce que vous devez savoir

Un Service fournit une IP stable et un DNS interne pour accéder aux pods. Il répartit le trafic entre tous les pods correspondant au sélecteur. Le type ClusterIP rend le service accessible uniquement depuis l'intérieur du cluster.

💡 Indice : Le selector du Service doit correspondre exactement aux labels des pods du Deployment.
kubectl get svc -n demo-app
kubectl get endpoints -n demo-app
8
Configurer un Ingress
Exposer l'application à l'extérieur du cluster via un nom de domaine.

🎯 Tâches à réaliser

  • Activer l'addon Ingress dans Minikube : minikube addons enable ingress
  • Créer un fichier k8s/ingress.yaml dans le namespace demo-app
  • Définir un host : demo.local
  • Router tout le trafic (/) vers le Service sur le port 80
  • Utiliser ingressClassName: nginx
  • Ajouter l'entrée dans /etc/hosts

📖 Ce que vous devez savoir

L'Ingress est un objet qui gère l'accès HTTP/HTTPS externe au cluster. Il utilise un contrôleur (nginx, traefik, etc.) pour router le trafic vers les services internes en fonction des règles (host, path).

💡 Indice : N'oubliez pas d'ajouter $(minikube ip) demo.local dans votre fichier /etc/hosts. Le pathType doit être Prefix.
kubectl get ingress -n demo-app
curl http://demo.local/health
9
Vérifier et tester le déploiement
S'assurer que tous les objets sont créés et que l'application fonctionne correctement.

🎯 Tâches à réaliser

  • Vérifier que tous les pods sont Running
  • Vérifier que le PVC est Bound
  • Tester /health, /version, /config
  • Créer une note via l'interface web ou l'API
  • Tester le login : cliquer sur "🔐 Connexion Admin" et entrer le mot de passe du Secret K8s
  • Vérifier la protection : essayer de supprimer une note sans être connecté (doit être bloqué)
  • Supprimer un pod et vérifier que la note persiste
# Vérification complète
kubectl get all -n demo-app
kubectl get pv,pvc -n demo-app
kubectl get configmap,secret -n demo-app
kubectl get ingress -n demo-app

# Test des endpoints
curl http://demo.local/health
curl http://demo.local/version
curl http://demo.local/config

# Test du login (utilise ADMIN_PASSWORD du Secret K8s)
curl -X POST http://demo.local/api/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"VOTRE_ADMIN_PASSWORD"}'

# Créer une note via API
curl -X POST http://demo.local/api/notes \
  -H "Content-Type: application/json" \
  -d '{"title":"Ma première note K8s","content":"Déployée avec succès!","category":"important"}'

# Tester la suppression sans token (doit échouer avec 401)
curl -X DELETE http://demo.local/api/notes/ID_NOTE

# Supprimer un pod et vérifier la persistance
kubectl delete pod -n demo-app -l app=k8s-demo --wait=false
kubectl get pods -n demo-app -w
curl http://demo.local/api/notes
